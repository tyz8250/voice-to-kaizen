# Issue #7-7 テスト技法の学習メモ

## 受講している講座

- Udemy「はじめてのソフトウェアテスト技法【全てのエンジニアが知るべき最重要テスト】」
- 状態：受講中

## 学ぶ目的

Issue #7-7でログインAPIのテストを行う前に、テストケースをどのような観点で考え、何を確認すればよいかを理解する。

単に`go test`を成功させるのではなく、次のことを説明できる状態を目指す。

- 何を保証するためのテストか
- 正常系と異常系をどう分けるか
- 入力値と期待結果をどう決めるか
- テストの抜け漏れをどう減らすか
- 単体テストと結合テストをどう使い分けるか

## Issue #7-7へつなげる観点

ログインAPIについて、まず次の形でテストケースを整理する。

```text
前提条件
↓
入力・操作
↓
期待するステータスコード
↓
期待するレスポンス
↓
確認したい仕様
```

現時点で想定している確認対象は次のとおり。

- 正しいemail・passwordでJWTが返る
- passwordが間違っている場合は`401 Unauthorized`になる
- emailが存在しない場合は`401 Unauthorized`になる
- JSONが不正な場合は`400 Bad Request`になる
- POST以外は`405 Method Not Allowed`になる
- DBを利用できない場合もpanicしない
- `JWT_SECRET`がない場合はJWTを発行しない
- JWTに`user_id`、`role`、`iat`、`exp`が含まれる

email・passwordの空欄チェックは、Issue #7-8で扱う予定とする。

## 講座で学んだこと

### 網羅性とピンポイント

テストでは、網羅性とピンポイントの両方が重要になる。

すべてを大量にテストするのではなく、テストの内容を具体化し、より少ないテストで多くのバグを発見できるケースを考える。

- 具体的な数値や入力値を当てはめる
- 境界や失敗しそうな箇所を狙う
- ときには意地悪な入力を使う
- 当たり前に動くと思っていることも、テストとして確認する

### 契約によるテスト

決められた仕様や入力範囲の中で、期待どおりに動くことを確認する。

ログインAPIでは、例えば次のような確認が当てはまる。

- 正しいemail・passwordでログインできる
- 間違ったpasswordを送ると`401 Unauthorized`になる
- 存在しないemailを送ると`401 Unauthorized`になる
- 成功時に仕様どおりのJWTが返る

### 防御によるテスト

決められた範囲外や、想定していない使われ方でも、システムが危険な状態にならないことを確認する。

ログインAPIでは、例えば次のような確認が考えられる。

- 壊れたJSONを送る
- POST以外のHTTPメソッドを使う
- DBへ接続できない状態でリクエストする
- `JWT_SECRET`を設定せずにログインする
- 想定外に長い文字列や特殊な文字を送る

### テスト設計に唯一の正解はない

どこまでテストするかに、常に同じ正解があるわけではない。

発生する可能性、バグが起きた場合の影響、テストを作成・維持するコストを見ながら、その都度コストに見合った判断をする必要がある。

テストを選ぶときは、次のことを考える。

- どのようなバグを発見したいか
- そのバグが起きると、どのくらい困るか
- 同じ目的のテストが重複していないか
- テストの作成と維持に、どのくらいコストがかかるか

Issue #7-7では、まずログインできないと機能全体を利用できなくなるケースと、認証に失敗しているのにJWTが発行される危険なケースを優先する。

### 同値クラステスト

テストを行っても同じ意味や結果になる入力を、同じグループとしてまとめる。

同じグループの値をすべて試すのではなく、代表する最小限のケースを選ぶ。テスト数を抑えながら、同じ種類の動作を確認できる。

ログインAPIでは、例えば次のように分けられる。

| 同値クラス | 代表するテストケース | 期待結果 |
| --- | --- | --- |
| 登録済みemail・正しいpassword | `admin@example.com` / `admin123` | `200 OK`とJWT |
| 登録済みemail・間違ったpassword | `admin@example.com` / `wrong-password` | `401 Unauthorized` |
| 未登録email | `unknown@example.com` / 任意のpassword | `401 Unauthorized` |
| JSONとして不正 | 壊れたJSONを1件 | `400 Bad Request` |
| POST以外のメソッド | `GET /login` | `405 Method Not Allowed` |

ただし、一見同じ結果になりそうでも、内部で異なる処理を通る場合は同じクラスとしてまとめてよいか慎重に考える。

### 境界値テスト

境界値テストでは、どこから結果が変わるのかを自分で明確に認識してからテストする。

一般的には、境界の直前、境界上、境界の直後を候補にする。ただし、仕様に境界が定義されていない状態で、テスト側だけが勝手に境界を決めないようにする。

現在のログインAPIでは、emailやpasswordの文字数制限がまだ決められていない。そのため、文字数の境界値テストを作る前に、まず許可する最小・最大文字数を仕様として決める必要がある。

JWTでは有効期限が24時間と決められているため、将来的には次の時点が境界になる。

- 有効期限の直前
- 有効期限と同じ時刻
- 有効期限を過ぎた直後

### 欠陥を一つずつ調べる

一つのテストが成功しても、その後に続く別の処理まで正しいとは限らない。

例えば、入力値の境界で期待どおりの結果になっても、その後のDB検索、パスワード照合、JWT生成が誤っている可能性がある。

```text
リクエストの受付
↓
JSONの変換
↓
DBからユーザーを検索
↓
bcryptでパスワードを照合
↓
JWTを生成
↓
レスポンスを返す
```

問題が起きたときは、この流れを一段階ずつ確認する。一つのテストへ多くの目的を詰め込みすぎず、「どの処理の、どの欠陥を見つけたいテストなのか」を明確にする。

### ドメイン分析テスト

ドメイン分析テストと、ON・OFF・IN・OUTという考え方を学んだ。

現時点では、分かったような、まだ分かっていないような状態。入力できる範囲を領域として捉え、境界や領域の内外から代表する値を選ぶ考え方だと理解しているが、それぞれの値をどのように選ぶかはまだ十分に説明できない。

#### 現時点での仮の理解

- ON：境界上の値
- OFF：境界のすぐ近くで、境界の開閉に応じて内側または外側から選ぶ値
- IN：有効な領域の内側にある代表値
- OUT：有効な領域の外側にある代表値

この整理は現時点での仮理解であり、実際のテストケースを作るときに再確認する。

OFFは常に範囲外とは限らない。境界値を有効として含む閉じた境界ではすぐ外側、境界値を含まない開いた境界ではすぐ内側から選ぶ、という違いがある。

これまでのテストでも、意図せず似た考え方を使っていた可能性がある。

例えば、「有効な値」「無効な値」「境界ちょうど」「境界を一つ超えた値」を試していた場合、それは明示的に名前を付けていなくても、ドメイン分析テストに近い考え方だったのかもしれない。

#### voice-to-kaizenで試せそうな場所

ログインAPIには、現時点でemail・passwordの文字数範囲が定義されていないため、ON・OFF・IN・OUTを明確に割り当てにくい。

一方、今後扱う`impact`、`urgency`、`effort`は1〜5という範囲が決められているため、ドメイン分析を試しやすそうに見える。

| 分類 | 値の候補 | 現時点での考え方 |
| --- | --- | --- |
| ON | `1`、`5` | 有効範囲の境界上 |
| OFF | `0`、`6` | 境界を一つ外れた値 |
| IN | `3` | 有効範囲の内側 |
| OUT | `-100`、`100` | 有効範囲から大きく外れた値 |

この表も仮の整理とし、講座の定義と照らし合わせてから正式なテストケースにする。

## 講座を見ながら追記すること

### 学んだ用語・技法

- 網羅性とピンポイント
- 契約によるテスト
- 防御によるテスト
- 同値クラステスト
- 境界値テスト
- ドメイン分析テスト
- ON・OFF・IN・OUT
- コストに見合ったテスト設計

### 理解できたこと

- テスト数を増やすこと自体が目的ではない
- 少ないテストで重要な不具合を発見できるケースを考える
- 仕様内の動作だけでなく、仕様外の入力に対する防御も確認する
- 当たり前だと思っている動作も、明示的に確認する必要がある
- 同じ意味になる入力は同値クラスとしてまとめ、代表値を選ぶ
- 境界値テストの前に、仕様上の境界がどこかを確認する
- 一つのテスト結果だけで、その後の処理まで正しいと判断しない
- 以前から無意識に、入力領域の内側・外側・境界を試していた可能性がある

### まだ分からないこと

- Issue #7-7で、どこまでを自動テストにするか
- 契約によるテストと防御によるテストを、それぞれ何件用意するか
- テストケースの優先順位を具体的にどう決めるか
- email・passwordの文字数や形式の境界をどのように決めるか
- ON・OFF・IN・OUTを、実際の仕様からどのように選ぶか
- OFFとOUTをどのように使い分けるか
- これまで行っていたテストのどれが、ドメイン分析テストに当たるのか

### voice-to-kaizenへ適用できそうなこと

- 正常系を1件だけ確認するのではなく、認証失敗時にJWTが発行されないことを重点的に確認する
- 似た結果になるケースを整理し、重複するテストを増やしすぎない
- APIの仕様外の入力でもpanicや情報漏えいが起きないことを確認する
- ログイン処理を段階に分け、どの処理の欠陥を調べるテストかを明確にする
- `impact`、`urgency`、`effort`の1〜5の範囲で、ドメイン分析テストを試す
